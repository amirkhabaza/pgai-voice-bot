# Architecture Document

## How the System Works

The voice bot is built on three main components: Twilio for telephony, Claude (claude-haiku) as the patient brain, and Twilio's built-in speech recognition for transcription. When a call is initiated via `main.py`, Twilio dials the target number and — once the call connects — sends a POST request to a locally running Flask server exposed to the internet via ngrok. The Flask server responds with TwiML (Twilio's XML instruction format) that tells Twilio what to say and how long to listen. On the first turn, Claude generates a realistic opening line based on the assigned patient persona. From there, the loop is: Twilio captures what the agent says using its `<Gather input="speech">` verb, sends the transcript to the `/agent_spoke` endpoint, Claude reads the full conversation history and generates the patient's next reply, and Twilio speaks it back using Amazon Polly's Joanna voice. This continues until Claude's response contains a goodbye signal, at which point the bot hangs up gracefully. All conversation turns are stored in memory keyed by CallSID and flushed to a JSON transcript file when the recording callback fires.

## Key Design Decisions

**Twilio's built-in STT over Deepgram streaming.** The original plan was to stream audio to Deepgram in real time, but Twilio's `<Gather speech>` approach is significantly simpler to implement reliably — it handles VAD (voice activity detection), silence detection, and audio buffering natively without requiring a bidirectional WebSocket connection. For a turn-based medical conversation (as opposed to a barge-in scenario), this is the right tradeoff: lower complexity, fewer failure modes, and fast enough for the use case. Deepgram would be the right upgrade if low-latency streaming or word-level timestamps were needed.

**Claude Haiku as the patient persona.** The patient brain needs to be fast (to keep conversational latency low), context-aware (to maintain persona across 15–20 turns), and capable of realistic improvisation when the agent says something unexpected. Haiku hits all three — it's the fastest Claude model, cheap enough to run across many calls without concern, and more than capable of maintaining a coherent patient persona. The system prompt defines a specific patient identity, goal, and set of realistic fake personal details, which grounds the conversation and makes it testable against specific scenarios. Full conversation history is passed on every turn so Claude always has complete context.

**Stateless Flask server with in-memory conversation store.** Rather than persisting conversation state to a database, conversations are stored in a Python dict keyed by CallSID for the duration of the call. This is appropriate for a testing tool where calls are short-lived and the primary output is the transcript file. The recording callback is used as the signal to flush the conversation to disk, ensuring the transcript is always complete before being written.
